# Baseline Evaluation Harness

This document describes the deterministic evaluator used to score the
prompt‑only baseline outputs against production‑aligned ground truth.
Unlike the analytics backend or the language model, the evaluator does
not infer or generate content; it applies a series of rule‑based
checks to determine whether a baseline prediction meets basic quality
and fidelity criteria.

## Purpose

After generating baseline predictions from the LLM harness, it is
important to understand how closely those predictions match the
expected outputs.  The evaluator compares each prediction against the
corresponding dataset row (via the `id` field) and produces both
per‑example records and an aggregate summary.  The evaluation covers:

1. **Schema compliance** — Does the prediction parse as JSON with
   exactly the four required keys (`executive_summary`, `main_finding`,
   `recommendations`, `confidence_reason`) and appropriate types?
2. **Faithfulness** — Does the prediction reuse only numbers that
   appear in the source report or the ground truth summary?  Numbers
   that are invented or unrelated are flagged.
3. **Recommendation grounding** — Are the predicted recommendations
   drawn from the set of allowed actions (from the source report’s
   `backend_recommendations` and the ground truth `recommendations`)?
4. **Confidence calibration** — Does the `confidence_reason` align
   with the dataset’s `confidence_label` (high, medium, low)?  Simple
   keyword heuristics detect mismatches.
5. **Product‑style checks** — Do the outputs respect basic length
   limits, avoid forbidden filler phrases, and limit the number of
   recommendations?

The evaluator is conservative by design.  It prefers to flag
potential issues rather than risk accepting ungrounded or stylistically
inappropriate content.

## Input Files

### Ground Truth Dataset

A JSONL file (e.g., `/mnt/data/valid(1).jsonl`) where each line
contains:

* `id`: unique identifier for the example.
* `domain`: optional domain label (sales, survey, etc.).
* `family_id`: optional grouping identifier.
* `confidence_label`: one of `high`, `medium`, or `low`.
* `source`: a dictionary containing the deterministic report, including
  `backend_executive_summary`, `key_findings`, `backend_recommendations`, and
  other metadata.  The evaluator uses these fields to determine
  allowed numbers and recommendation actions.
* `target`: the human‑written canonical output with the four keys used
  for training and evaluation.

### Baseline Predictions

A JSONL file generated by the baseline inference harness (see
`BASELINE_INFERENCE.md`) where each line contains:

* `id`, `domain`, `family_id`, etc.  These fields should match those
  from the ground truth.
* `raw_response`: the raw model output.
* `parsed_output`: the parsed JSON object from the model (or null if
  parsing failed).
* `parse_success`: boolean flag indicating whether parsing succeeded.
* `parse_error`: error message if parsing failed.
* Additional metadata fields (backend, model, prompt_length).

The evaluator matches rows by `id`.  If a prediction is missing for a
dataset row, it is treated as a parse failure.

## Checks Performed

### Schema Validation

The evaluator ensures that `parsed_output` contains exactly the four
expected keys and that the values have the correct types (strings and
list of strings).  Any deviation (missing key, extra key, wrong type)
fails the schema check.

### Faithfulness (Number Grounding)

Numeric tokens (e.g., `34`, `2.74M`, `88%`, `0.984`) in the
prediction must also appear in either the source report or the target
summary.  This prevents the model from inventing new numbers.  The
evaluator extracts numbers from common text fields in the source and
target and compares them against the prediction.

### Recommendation Grounding

The model’s recommended actions must be drawn from the allowed action
set.  Allowed actions are the `backend_recommendations` from the
source report and the ground truth `recommendations`.  A prediction
passes if each recommended sentence contains (or is contained in) one
of the allowed actions, ignoring case.

### Confidence Calibration

The evaluator checks the tone of the `confidence_reason` against the
ground truth `confidence_label`:

* **High** — The reason must contain language indicating high
  confidence (e.g., "high confidence", "strong", "verified") and must
  not contain cautionary language (e.g., "uncertain", "limited").
* **Low** — The reason must include cautionary language and must not
  proclaim high confidence.
* **Medium** — The reason should not strongly indicate high or low
  confidence; moderate or neutral language is acceptable.

These checks use simple keyword heuristics and may be extended later.

### Product‑Style Checks

To encourage concise, product‑like responses, the evaluator enforces:

* Maximum word counts for each field (summary ≤ 60 words, main finding
  ≤ 40 words, confidence_reason ≤ 40 words, each recommendation ≤ 30
  words).
* At most three recommendations.
* No forbidden filler phrases such as "In conclusion", "As a language
  model", "Overall" or first‑person chatty language.

Any violation results in a style failure.

## Output Files

### Per‑Example Evaluation (`--output`)

Each line contains a JSON object with the fields:

* `id`, `domain`, `family_id`
* Booleans for `parse_success`, `exact_key_match`, `schema_valid`,
  `faithfulness_pass`, `recommendation_grounding_pass`,
  `confidence_calibration_pass`, `style_pass`, `overall_pass`
* `issue_flags`: a list of short tags describing why a row failed.

Rows that fail any check will have `overall_pass` set to `false`.

### Aggregate Summary (`--summary`)

Contains overall counts and rates for each check, domain counts, and
failure reason histograms.  This summary helps track baseline
performance across different domains and metrics.

## Running the Evaluator

Use the `scripts/run_baseline_evaluation.py` script.  A typical
invocation on the validation split and mock predictions:

```bash
python scripts/run_baseline_inference.py \
  --input /mnt/data/valid(1).jsonl \
  --output baseline_valid_mock.jsonl \
  --backend mock \
  --max-samples 5

python scripts/run_baseline_evaluation.py \
  --dataset /mnt/data/valid(1).jsonl \
  --predictions baseline_valid_mock.jsonl \
  --output baseline_valid_eval.jsonl \
  --summary baseline_valid_eval_summary.json
```

Inspect the summary JSON to understand which checks most often fail and
to prioritise improvements to the prompt or model.  Do **not** use
`test(1).jsonl` for tuning; reserve it for final evaluation.
