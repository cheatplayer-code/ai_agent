# Prompt‑Only Baseline Experiment Results

This document summarises the prompt‑only baseline experiments on the
validation split of the production‑aligned dataset (`valid.jsonl`).  The
goal of these experiments was to compare different prompt
formulations using the existing baseline inference harness and
deterministic evaluator, **without any model fine‑tuning**.  The
baseline prompts instruct a language model to produce a JSON object
with four keys: `executive_summary`, `main_finding`,
`recommendations`, and `confidence_reason`.

## Prompt Variants Tested

Four prompt variants were defined in `src/llm_baseline/prompt_variants.py`:

1. **strict_json_minimal** – concise instructions, emphasising JSON output
   and faithfulness.
2. **strict_json_grounded** – adds stronger anti‑hallucination guidance
   and cautions against inventing numbers, entities, or causes.
3. **strict_json_grounded_fewshot** – includes the same guidance as
   `strict_json_grounded` plus a single lightweight few‑shot example
   demonstrating the desired input and output structure.
4. **strict_json_ultra_conservative** – prioritises caution, asking
   the model to reflect uncertainty and avoid over‑interpretation.

The `run_prompt_baseline_experiments.py` script runs inference for
each variant, evaluates the outputs against the ground‑truth reports
using the deterministic evaluator, and aggregates summary metrics.

## Backend and Setup

Due to the execution environment limitations, experiments were run
using the **mock** backend.  This backend does not invoke a real
language model; instead it returns a trivial summary derived from the
first line of the report.  As such, the results reported below are
not indicative of real model performance but demonstrate the
end‑to‑end harness and evaluation flow.  To run these experiments with
an actual model (e.g., a Hugging Face instruct model) on a
GPU‑enabled environment, invoke the experiment script with
`--backend transformers --model <model_id>`.

The experiments processed the first **five** examples of the
validation split (`--max-samples 5`) for a quick smoke test.  For
production benchmarking, the full validation set should be used.

## Best Variant

For this small mock run, all variants achieved identical metrics.  In
the absence of meaningful differences, the comparison script selects
the lexicographically first variant (`strict_json_grounded`) as the
best variant by overall pass rate, faithfulness pass rate, and
confidence calibration rate.  This selection is arbitrary due to
metric ties.

## Interpretation and Next Steps

Even under trivial summarisation, the evaluator flagged **faithfulness
and confidence calibration** as the primary failure reasons.  The
overall pass rate across all variants was 0%, illustrating that a
prompt‑only approach without a trained model fails to meet the
expected quality bar.  These results underscore the need for:

1. **Running with a real language model** – The mock backend cannot
   evaluate the actual effectiveness of different prompts.  Running the
   same experiment with a base instruct model (e.g., GPT‑2 or a
   lightweight open‑source instruct model) is necessary to obtain
   actionable insights.
2. **Fine‑tuning or adapter training** – Given the complexity of the
   task (summarising structured analysis reports into concise product
   language), it is likely that prompt‑only performance will be
   insufficient.  Fine‑tuning or low‑rank adaptation (LoRA) on
   representative examples may be required to achieve acceptable
   quality, particularly for faithfulness and confidence calibration.
3. **Further prompt engineering** – Once a base model is in place,
   additional prompt variations, better few‑shot examples, or more
   domain‑specific instructions could help improve model outputs.

In summary, the prompt‑only baseline experiments on the mock backend
demonstrate the mechanics of the inference and evaluation pipeline but
do not yield meaningful quality improvements.  Running the same
experiments with a real model and iterating on prompt design and
fine‑tuning will be crucial in the next phase.
