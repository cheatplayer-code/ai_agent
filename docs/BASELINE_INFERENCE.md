# Baseline Inference Harness

This document explains the purpose and usage of the **baseline inference harness**
for the deterministic backend.  The harness is a lightweight wrapper
around the frozen analytics output that will eventually feed a large
language model (LLM) for more fluent reporting.  At this stage,
**no fine‑tuning or model training occurs**—the goal is to assemble
prompts, invoke an inference backend, parse the model’s response, and
capture metadata for later evaluation.

## Purpose

The deterministic backend produces rich analysis reports.  The baseline
inference harness takes these reports (or slices thereof) as input via
the `source` field in a JSONL dataset and asks a language model to
generate four pieces of information:

1. **executive_summary** — A concise overview of the analysis and key
   insights.
2. **main_finding** — The single most important quantified insight.
3. **recommendations** — A list of 1–3 succinct, actionable steps
   supported by the report.
4. **confidence_reason** — A brief explanation of why the output is
   confident or cautious, tied to the report evidence.

These four fields constitute the baseline output schema.  They are
produced by the LLM and will be compared against human references in
later evaluation phases.  Other report fields (such as charts or DQ
issues) remain the domain of the deterministic backend and **must not
be generated by the LLM**.

## Input Schema

The harness expects an input JSONL file in which each line is a JSON
object with at least a `source` field containing the text of an
analysis report.  Additional fields such as `id`, `domain`,
`family_id`, or `split` are preserved and attached to the output
metadata.  An example input line:

```json
{
  "id": "example-001",
  "domain": "survey",
  "family_id": "fam-123",
  "split": "valid",
  "source": "The dataset is primarily numeric ... (full deterministic report)"
}
```

## Prompt Construction

Prompts are built using a configurable prompt builder in
`src.llm_baseline.prompt_variants`.  The default builder
(`strict_json_minimal`) constructs a prompt that instructs the model
to stay faithful to the source report, forbids invention of numbers or
entities, limits the number of recommendations, and demands a JSON
response with exactly four keys.  Other variants (e.g.,
`strict_json_grounded`, `strict_json_grounded_fewshot`, and
`strict_json_ultra_conservative`) provide stronger grounding language
or a few‑shot example.  The raw report text is always appended under
a `Report:` heading.  You can select a variant via the CLI with
`--prompt-variant`.

## Supported Backends

Currently two backends are supported via `src.llm_baseline.inference.load_backend()`:

1. **mock** — Returns a trivial JSON output using the first line of
   the report as both the executive summary and main finding.  Use this
   for unit tests and smoke checks when no model is available.
2. **transformers** — Uses a Hugging Face causal language model (e.g.,
   GPT‑2) for local inference.  Requires the `transformers` library and
   may need a GPU for reasonable speed.  Generation parameters such as
   `max_new_tokens`, `temperature`, and `top_p` are configurable.

Additional backends (e.g., via API calls) can be implemented later
without changing the harness interface.

## Output Schema

The harness writes a JSONL file where each line contains:

- `id`, `domain`, `family_id`, `split` — Passed through from the input or
  the CLI `--split` argument.
- `raw_response` — The raw string produced by the model backend.
- `parsed_output` — A dictionary with the four required keys if
  parsing succeeded, otherwise null.
- `parse_success` — Boolean indicating whether parsing was successful.
- `parse_error` — Error message when parsing fails.
- `backend` — Name of the backend used (e.g., `mock`, `transformers`).
- `model` — Model identifier for the transformers backend (null for mock).
- `prompt_length` — Length of the prompt in characters (for analysis).

These fields ensure that downstream evaluators have sufficient context
to score the outputs and diagnose errors without rerunning inference.

## Running the Harness

Use the `scripts/run_baseline_inference.py` script to run inference.  A
minimal invocation on a few validation examples might look like this:

```bash
python scripts/run_baseline_inference.py \
  --input /path/to/valid.jsonl \
  --output baseline_valid_mock.jsonl \
  --backend mock \
  --max-samples 5
```

To run using a local Hugging Face model (assuming it is installed):

```bash
python scripts/run_baseline_inference.py \
  --input /path/to/valid.jsonl \
  --output baseline_valid_gpt2.jsonl \
  --backend transformers \
  --model gpt2 \
  --max-samples 3 \
  --device cpu
```

**Note:** Running a language model may be resource‑intensive.  The
mock backend should be used for unit tests and quick sanity checks.

## What This Step Does Not Do

- No fine‑tuning or training of language models.  Models are used
  purely for inference.
- No evaluation or scoring.  The harness merely collects responses and
  parses them; evaluation will be implemented in a later stage.
- No changes to the deterministic backend.  The backend remains the
  authoritative source for analysis.  Only the `source` field of
  prepared datasets is passed into the LLM.

## Reference

This harness is part of a multi‑stage pipeline.  It comes after the
backend v1 freeze (documented in `docs/BACKEND_V1_FREEZE.md`) and
before training or evaluating any LLM models.
